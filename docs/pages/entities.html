Entity System

ECS-based entity management with bevy_ecs 0.15, mob AI, and spawn systems.

ECS Architecture

MC-RS uses bevy_ecs 0.15 (the standalone ECS library, not the full Bevy game engine) as its entity-component-system foundation. The GameWorld struct wraps a bevy_ecs::World and provides a high-level interface for entity management, system execution, and game state queries.

Components

Entities are composed of lightweight data components that are added and removed dynamically:

  • Position — 3D world coordinates (Vec3: x, y, z)
  • Velocity — Per-tick movement vector, affected by gravity and AI
  • Health — Current and maximum hit points
  • MobType — Enum identifying the mob species (Zombie, Skeleton, Cow, etc.)
  • BehaviorList — Vec<Box<dyn Behavior>> defining the AI behavior stack
  • Target — Currently targeted entity (for hostile mobs)
  • SpawnedBy — Tracks which player or system spawned the entity

Systems

ECS systems run in a defined order each game tick (20 TPS, every 50ms):

  1. Gravity — Applies downward velocity to all entities not on ground
  2. AI tick — Evaluates and executes mob behaviors based on priority
  3. Movement — Applies velocity to position, performs collision detection
  4. Spawn/Despawn — Natural mob spawning and distance-based despawning
  5. Death — Removes entities at zero health, triggers death events
// GameWorld tick pipeline (simplified)
impl GameWorld {
    pub fn tick(&mut self, current_tick: u64) {
        self.system_gravity();         // Apply gravity to entities
        self.system_ai_tick();         // Run AI behaviors
        self.system_movement();        // Apply velocity, collisions
        self.system_spawn_despawn();   // Natural mob lifecycle
        self.system_death();           // Remove dead entities
    }
}
bevy_ecs note: Since World::query() in bevy_ecs 0.15 requires &mut World, all methods that use queries take &mut self. This is a consequence of using the standalone ECS library outside the full Bevy scheduler.

Mob Types

MC-RS implements 5 mob types through the MobRegistry, which provides static definitions for each species including health, damage, drops, and default behaviors.

Each mob is spawned as an ECS entity with its type-specific components. The /summon command can spawn any registered mob type at a specified position.

AI Behaviors

Mob AI uses a priority-based behavior system. Each mob has a BehaviorList component containing a Vec<Box<dyn Behavior>> sorted by priority (lower number = higher priority). Every tick, the AI system evaluates behaviors from highest to lowest priority.

Behavior Types

Behaviors are categorized into three types that determine execution rules:

  • Movement — Only one movement behavior can be active at a time. If a higher-priority movement behavior activates, the current one is interrupted.
  • Passive — Multiple passive behaviors can run simultaneously alongside any active movement behavior.
  • TargetSelector — Only one target selector can be active at a time. Determines which entity the mob targets for attacks.

Behavior Table

PriorityBehaviorTypeDescription
0 Float Passive Keeps the mob at the water surface when in water. Applies upward velocity to prevent drowning. Always active when submerged.
1 NearestAttackableTarget TargetSelector Scans for the nearest player within detection range. Sets the mob's attack target. Used by hostile mobs (zombies, skeletons) to acquire targets.
1 Panic Movement Activated when a passive mob takes damage. The mob runs in a random direction at increased speed for several seconds, fleeing from the attacker.
2 HurtByTarget TargetSelector Retargets the mob's attack to whatever entity last dealt damage to it. Provides aggro response for hostile mobs.
2 MeleeAttack Movement Moves toward the current target and deals melee damage when within reach. Handles attack cooldown and pathfinding to the target position.
3 TemptGoal Movement The mob follows a nearby player who is holding a tempting item (wheat, carrot, or seeds depending on mob type). Used for leading animals.
4 BreedGoal Movement When in love mode, seeks another same-type mob also in love mode. Moves toward the partner and triggers breeding when close enough.
5 RandomStroll Movement Wanders randomly when idle. Selects a random nearby position and walks toward it. Uses flat pathfinding to navigate around obstacles.
6 LookAtPlayer Passive Rotates the mob's head to face the nearest player within a short range. Provides visual feedback that mobs are aware of players.
SpatialGrid: AI queries like "find nearest player" use a SpatialGrid data structure for O(1) lookups instead of iterating all entities. This is critical for performance when many mobs are active simultaneously.

Natural Spawning

Every 100 game ticks (5 seconds), the spawn system checks if the current mob count is below the cap for each category. If so, it selects random positions within a radius around each player and attempts to spawn appropriate mob types. The spawn position must be on a solid block with sufficient space above for the mob's hitbox.

Despawning

Mobs are removed when they are more than 128 blocks from all connected players. This check runs each spawn cycle. Mobs spawned by the /summon command follow the same despawn rules as naturally spawned mobs.

Breeding System

Passive mobs (cow, pig, chicken) can be bred by players to produce baby mobs.

TemptGoal

Each breedable mob type is attracted to a specific item held by a nearby player:

  • Cow — Wheat
  • Pig — Carrot
  • Chicken — Seeds (wheat seeds, beetroot seeds, melon seeds, pumpkin seeds)

When a player holds the tempting item, nearby mobs of the matching type will follow the player at walking speed. The HeldItemName is synced from the PlayerConnection to the ECS so behaviors can check it. Feeding a mob its tempt item (via UseItemOnEntity interaction) puts the mob into love mode.

BreedGoal

When two mobs of the same type are both in love mode and within proximity:

  1. They move toward each other using the BreedGoal movement behavior
  2. When close enough, a baby mob is spawned at their midpoint
  3. Both parents exit love mode and enter a 5-minute breeding cooldown
  4. Love particles are emitted via LevelEvent packets during love mode and at the moment of breeding

Baby mobs have a smaller hitbox and grow into adults over time.

Player Mirror Entities

Each connected player has a corresponding mirror entity in the ECS world. This entity exists so that the ECS systems (AI targeting, spatial queries, collision) can interact with players uniformly alongside mobs.

  • Spawned on connect — When a player joins the server and enters the Playing state, a mirror entity with Position and Health components is created in the GameWorld.
  • Position sync — The update_player_position function runs each tick, copying the authoritative position from PlayerConnection.position into the ECS Position component. This ensures AI behaviors (like NearestAttackableTarget) see accurate player positions.
  • Despawned on disconnect — When a player disconnects, their mirror entity is removed from the ECS world.
  • Not the source of truth — The PlayerConnection struct remains the authoritative source for all player state (position, health, inventory, effects). The mirror entity is a read-only projection used by ECS queries.
Design note: Player mirror entities exist because the ECS needs to query players alongside mobs for AI targeting and spatial operations. Without them, every AI system would need special-case code to check both the ECS world and the separate PlayerConnection map.

Network Packets

Entity state is synchronized to clients using three primary packets. All entity packets use a runtime_id (VarUInt64) to identify entities uniquely within the session.

AddActor (0x0D)

Sent when a new non-player entity spawns within a player's view distance. Contains all data needed for the client to create the entity.

AddActor {
    unique_entity_id: VarLong,     // Persistent entity ID (signed zigzag)
    runtime_entity_id: VarUInt64,  // Session-unique runtime ID
    entity_type: String,           // e.g., "minecraft:zombie"
    position: Vec3,                // 3x f32_le (x, y, z)
    velocity: Vec3,                // 3x f32_le (motion)
    pitch: f32_le,                 // Vertical rotation
    yaw: f32_le,                   // Horizontal rotation
    head_yaw: f32_le,              // Head rotation
    body_yaw: f32_le,              // Body rotation
    attributes: [Attribute],       // Health, movement speed, etc.
    metadata: [MetadataEntry],     // Entity flags, name, variant, etc.
    links: VarUInt32,              // 0 (entity links, unused)
}

MoveActorAbsolute (0x10)

Sent each tick for entities that have moved, providing their new absolute position and rotation.

MoveActorAbsolute {
    runtime_entity_id: VarUInt64,  // Which entity moved
    flags: u16_le,                  // Bit flags (on_ground, teleport)
    position: Vec3,                 // New absolute position
    pitch: u8,                      // Compressed: angle * 256 / 360
    yaw: u8,                        // Compressed rotation
    head_yaw: u8,                   // Compressed head rotation
}
Angle compression: Rotation angles in MoveActorAbsolute are compressed from floating-point degrees to a single byte each. The formula is (angle * 256.0 / 360.0) as u8, giving approximately 1.4 degrees of precision per step. This reduces bandwidth by transmitting 3 bytes instead of 12 for rotations.

RemoveActor (0x0E)

Sent when an entity is despawned, killed, or moves out of view distance. The client destroys the entity and frees its resources.

RemoveActor {
    unique_entity_id: VarLong,  // The entity to remove (signed zigzag)
}

Entity visibility is managed per-player: when a mob moves into a player's loaded chunk area, an AddActor is sent. When it moves out, a RemoveActor is sent. This ensures clients only track entities they can actually see, keeping memory and rendering costs bounded.
