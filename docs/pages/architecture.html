Architecture

12-crate workspace designed for modularity and clean separation of concerns.

Crate Dependency Diagram

The workspace is organized as a directed acyclic graph of crate dependencies. mc-rs-server sits at the top as the entry point, depending on all other crates.

Crate Details

mc-rs-server

The server entry point and main binary crate. Hosts the tokio async runtime, the ConnectionHandler state machine that manages each client's lifecycle, and the game tick loop running at 20 TPS (50ms intervals via tokio::select!). Orchestrates all other crates: receives raw packets from RakNet, decrypts, decodes, dispatches to game logic, and broadcasts responses back to clients.

mc-rs-proto

Implements 50+ Bedrock packet types with manual ProtoEncode/ProtoDecode implementations (no derive macros). Provides the VarInt/VarLong zigzag codec, VarUInt32/VarUInt64, zlib compression, and batch framing. Large payloads like biome definitions and entity identifiers are embedded via include_bytes!.

mc-rs-raknet

Full RakNet UDP transport implementation. Handles reliability layers (reliable, reliable-ordered, unreliable), packet fragmentation and reassembly, sequence/ordering channels, acknowledgments (ACK/NAK), and congestion control. This is the lowest-level network layer that sits between raw UDP sockets and the protocol codec.

mc-rs-crypto

Provides ECDH P-384 key exchange for the encryption handshake, AES-256-CFB8 stream cipher for ongoing packet encryption, and JWT verification (via the jsonwebtoken crate) for validating Xbox Live authentication chains during login.

mc-rs-nbt

Implements the NBT (Named Binary Tag) serialization format in both little-endian (disk) and network (protocol) variants. Supports all tag types: compound, list, byte/int/long arrays, strings, and numeric types. Used by the protocol layer for block entity data, item NBT, and metadata blobs.

mc-rs-world

Manages chunk storage, the block registry (using FNV-1a hashes for network IDs), Perlin noise terrain generation (Overworld with 10 biomes, Nether, End), and LevelDB persistence (via rusty-leveldb). Handles sub-chunk encoding, biome serialization, and the block tick scheduler for fluids, gravity, and redstone.

mc-rs-game

The gameplay logic crate, built on bevy_ecs 0.15 (standalone, no full Bevy engine). Contains the ECS GameWorld, combat system (armor reduction, enchantment effects, critical hits, potion effects), food/hunger/saturation mechanics, the RecipeRegistry (~50 recipes), EnchantmentRegistry (37 enchantments), mob AI (9 behaviors with priority system), and XP formulas.

mc-rs-command

Command parsing framework with typed arguments (int, float, string, position, target selector). Implements entity selectors (@a, @p, @r, @e, @s) with filter predicates. The CommandRegistry handles the AvailableCommands packet for client-side tab completion.

mc-rs-plugin-api

Defines the Plugin trait and ServerApi interface. Provides 15 event types (PlayerJoin, PlayerLeave, PlayerChat, BlockBreak, BlockPlace, etc.) and the PendingAction queue for plugins to schedule server-side effects (send messages, kick players, show forms, teleport).

mc-rs-plugin-lua

Lua scripting runtime powered by mlua. Provides host function bindings that mirror the Rust plugin API, allowing Lua scripts to register event handlers, send messages, manipulate inventories, and interact with the game world. Supports hot-reload via the /reload command.

mc-rs-plugin-wasm

WASM plugin runtime powered by wasmtime. Plugins compiled to WebAssembly run in a fully sandboxed environment with controlled access to server APIs via imported host functions. Supports manifest.json for plugin metadata and permissions.

mc-rs-behavior-pack

Behavior pack loader that parses JSON definitions for entities, items, blocks, recipes, and loot tables. Extends the built-in registries at startup, allowing server operators to add custom content without writing code. Configured via the [packs] section in server.toml.

Data Flow

Every packet follows a consistent pipeline through the server:

Inbound: Client → UDP Socket → RakNet (reassemble, order) → Decrypt (AES-256-CFB8) → Decompress (zlib) → Batch Split → Packet Decode → ConnectionHandler → Game World (ECS)
Outbound: Game World (ECS) → OutgoingEvents → ConnectionHandler → Packet Encode → Batch → Compress (zlib) → Encrypt (AES-256-CFB8) → RakNet (fragment, reliability) → UDP Socket → Client

The ConnectionHandler is a state machine with well-defined states:

  1. Handshake — Waiting for RequestNetworkSettings
  2. Login — JWT verification, encryption negotiation
  3. ResourcePacks — Resource pack exchange
  4. PreSpawn — StartGame, chunk loading, inventory sync
  5. Playing — Full gameplay (movement, combat, crafting, commands)

Each state only accepts its valid set of packet types. Any out-of-state packet is rejected, preventing protocol exploits.

Threading Model

MC-RS uses a single-task async model built on tokio:

  • Network I/O — A single tokio task handles all UDP socket reads/writes via the RakNet layer. Each client connection is a ConnectionHandler struct (not a separate task), processed cooperatively.
  • Game Tick — The game world is ticked synchronously at 20 TPS (50ms intervals) using tokio::select! alongside packet processing. This avoids the complexity of cross-thread ECS access.
  • Chunk Generation — Computationally expensive terrain generation is offloaded to spawn_blocking with Arc-wrapped generators, running in parallel on tokio's blocking thread pool.
  • Persistence — LevelDB writes (chunk saves, player data) happen on the main task during auto-save intervals or server shutdown, since rusty-leveldb::DB is !Send.
  • Console/RCON/Query — Each runs in its own tokio task: stdin reader for console, TCP listener for RCON (Source protocol), and UDP listener for Query (GameSpy4).
Design choice: The single-task model was chosen over per-connection tasks to avoid the complexity of synchronizing shared game state across threads. With 20 TPS and efficient packet batching, a single task comfortably handles dozens of concurrent players.
